# 调度延迟诊断与调优实战

## 前言

**C：** 知识点铺垫完了，这一章我们干一件事：把一条真实的调度延迟问题从头排到尾。工具箱里有 `/proc/schedstat`、`trace-cmd`、`perf sched`、`bpftrace`、`cyclictest`、`runqlat`；方法上从"宏观看一眼"到"定位到具体 CPU/任务/锁"一步步收敛。最后给出一份可落地的调优 checklist。

## 先明确我们在追什么

常见的"延迟"其实是若干不同问题的总称，搞清楚是哪一类，排查方向完全不同：

| 延迟类型 | 含义 | 常用工具 |
|----------|------|----------|
| 唤醒延迟 | 任务被唤醒 → 真正开始跑 | `runqlat`、`perf sched latency` |
| 抢占延迟 | 高优先级任务 ready → 低优任务让位 | ftrace `preemptirqsoff` |
| 调度延迟 | 时间片到期 → 切换完成 | `perf sched timehist` |
| 中断延迟 | IRQ 到达 → ISR 开始 | ftrace `irqsoff`、`cyclictest` |
| 尾延迟（throttle） | 某个百分位的 request 特别慢 | `cpu.stat`、BCC `cpudist` |

**优先回答一个问题：你看到的是"平均变慢"还是"偶发尖刺"？** 前者多半是 CPU 不够（或被限流），后者多半是某个调度事件卡了。

## 快速体检清单

遇到性能问题，先跑一遍这几条命令，5 分钟内能把方向定下来。

### 1. CPU 负载与上下文切换

```bash
vmstat 1 10
# r: 可运行任务数, 大于 CPU 数 = CPU 饥饿
# cs: 每秒上下文切换, 异常高说明争锁或唤醒风暴
# us/sy/id/wa: 用户/系统/空闲/iowait
```

### 2. 每颗 CPU 的压力

```bash
mpstat -P ALL 1 10
# 单核满载 + 其它核空 = 负载均衡有问题
# %sys 偏高 = 锁争用或大量中断
# %soft 偏高 = 软中断积压
```

### 3. 运行队列延迟

```bash
sudo runqlat-bpfcc 5 3
# 直方图显示"从 ready 到 on-cpu"的分布
# us 级均值 = 正常; ms 级尾部 = 有问题
```

### 4. 系统整体压力（PSI）

```bash
cat /proc/pressure/cpu
# some avg10=... avg60=... avg300=... total=...
# some: 有任务因 CPU 而延迟
# full: 所有任务都在等 CPU
```

PSI（Pressure Stall Information）是现代 Linux 上最直观的"CPU 压力"指标。avg10 超过 10% 就值得警觉。

### 5. cgroup throttle

```bash
find /sys/fs/cgroup -name cpu.stat -exec sh -c \
  'echo "==> $0"; grep -E "nr_throttled|throttled_usec" $0' {} \;
```

一分钟内 `nr_throttled` 增加意味着该 cgroup 被 CPU quota 限流。

## 深入：perf sched 的典型用法

`perf sched` 是定位调度问题的第一工具。

### record：采集一段调度事件

```bash
sudo perf sched record -a -- sleep 30
```

30 秒后会生成 `perf.data`（通常几百 MB）。生产上建议先用 `-C 0,1`（只采样特定 CPU）或 `-p <pid>`（只跟踪特定进程）减小数据量。

### latency：按任务聚合

```bash
sudo perf sched latency --sort max
#            Task          |   Runtime ms | Switches |  Avg delay ms | Max delay ms |
# --------------------------------------------------------------------------------
#  kworker/u8:2:1234       |     12.345   |      567 |         0.012 |       34.567
#  redis-server:2345       |    156.789   |    12345 |         0.003 |        0.256
```

`Max delay` 是关键——这个任务最长一次"从 ready 到 on-cpu"等了多久。超过 10ms 就说明要细查了。

### map：可视化切换

```bash
sudo perf sched map
```

输出类似：

```
 *A0             1234.567: sched_switch
  A0 *B0         1234.568: sched_switch
  A0  B0 *C0     1234.569: sched_switch
```

每列是一颗 CPU，`*` 是当前跑的任务。能直观看到"C0 为什么一直没被调度"。

### timehist：按时间线

```bash
sudo perf sched timehist | less
#       time    cpu  task name           wait time  sch delay   run time
#                    [tid/pid]              (msec)     (msec)     (msec)
# 1234.567     [01]  redis-server         1.234      0.012      5.678
```

`wait time` = 睡了多久，`sch delay` = 唤醒到运行的延迟，`run time` = 这次跑了多久。这个视图最适合看"一个 request 具体经历了什么"。

## 用 bpftrace 精确定位

当 `perf sched` 告诉你"某个任务延迟大"，下一步是用 bpftrace 找出"它在等什么"。

### 谁在唤醒谁

```bash
sudo bpftrace -e '
tracepoint:sched:sched_wakeup
/str(args->comm) == "redis-server"/
{ @wakers[comm] = count(); }'
```

跑一段时间后 Ctrl-C，看哪些进程在唤醒 redis-server，是否符合预期（客户端 epoll 线程唤醒 vs. 一堆别的东西打扰）。

### 运行队列长度分布

```bash
sudo bpftrace -e '
tracepoint:sched:sched_stat_runtime
{ @runtime = hist(args->runtime); }'
```

### 某个 CPU 的长时间无切换

```bash
sudo bpftrace -e '
tracepoint:sched:sched_switch
{
    $delta = nsecs - @last[cpu];
    @gap = hist($delta);
    @last[cpu] = nsecs;
}'
```

超长尾（>10ms 的 gap）就是"这颗 CPU 被卡住了 10ms"。

### 跟踪 cond_resched 缺失

如果怀疑内核里有长循环没 `cond_resched`：

```bash
sudo bpftrace -e '
kprobe:schedule
{ @stack[kstack()] = count(); }' | head -50
```

看看 `schedule()` 的调用栈分布，异常的调用路径会暴露出来。

## 实战案例 1：容器周期性卡顿

**现象**：K8s pod 内的服务每 100ms 都会"卡"一下，P99 延迟周期性尖刺。

**第一步**：看 cgroup stat：

```bash
cat /sys/fs/cgroup/kubepods.slice/.../cpu.stat
# nr_throttled 12345
# throttled_usec 1234567
```

`nr_throttled` 在涨，基本锁定 CFS bandwidth 限流。

**第二步**：确认 quota：

```bash
cat /sys/fs/cgroup/.../cpu.max
# 50000 100000  <- 0.5 CPU per 100ms
```

**第三步**：看实际使用：

```bash
# 瞬时 CPU 使用可能 burst 到 1.2 核, 超出了 0.5 的限制
```

**解决方案**（按偏好排序）：

1. 把 quota 改成 `max` 或提到实际需要的 2 倍；
2. 使用 `cpu.weight` 替代 `cpu.max`（软限制）；
3. 调大 period（如果内核 >= 5.14，可以配合 `cpu.max.burst` 允许 burst）；
4. 检查应用是不是本来就有 thundering herd，从源头降 burst。

**误区**：不要简单地把 `cpu.max` 放大到"看起来没 throttle 就行"，真正的容量规划要看实际的 `cpu.stat.usage_usec` 增长速率。

## 实战案例 2：RT 任务偶发大延迟

**现象**：一个 SCHED_FIFO 音频线程每隔几分钟延迟突破 10ms，用户听到爆音。

**第一步**：看 RT throttling：

```bash
dmesg | grep -i throttle
# [...] sched: RT throttling activated
```

确认是 `sched_rt_runtime_us` 触发。

**第二步**：确认配置：

```bash
sysctl kernel.sched_rt_runtime_us
# 950000
```

意味着 95% 的时间给 RT，但如果你的 RT 任务总时间超过 95%，就会被整体 throttle 5%。

**第三步**：调整或转 DL：

- 如果确实需要接近 100% 的 RT，改 `kernel.sched_rt_runtime_us = -1`（**有风险！**）；
- 更好的做法是转成 `SCHED_DEADLINE`，它有**每任务 CBS**而不是全局限流；
- 或者减少 RT 工作量，让其稳定在 <95%。

**第四步（如果还有尾延迟）**：检查中断和软中断：

```bash
sudo cat /proc/interrupts | sort -k2 -n | tail -10
# 看哪颗 CPU 接中断最多

sudo mpstat -P ALL 1
# %soft 瞬间升到 40%+? 软中断压到了 RT 任务跑的核
```

解决：`irqaffinity=0-1`、`taskset -pc 2,3 <audio_pid>` 把中断和 RT 任务分开。

## 实战案例 3：负载均衡打乱缓存

**现象**：某个数据库进程单核性能很好，多核部署后反而更慢。

**第一步**：看任务迁移频率：

```bash
sudo bpftrace -e '
tracepoint:sched:sched_migrate_task
/str(args->comm) == "mysqld"/
{ @[args->orig_cpu, args->dest_cpu] = count(); }'
```

如果迁移数百/秒，说明 CFS 在不停"帮你搬家"。

**第二步**：看 cache miss：

```bash
sudo perf stat -e cache-misses,cache-references -p <pid> sleep 10
# miss ratio > 10% 就值得优化
```

**第三步**：用 taskset 绑核：

```bash
taskset -pc 0-7 <mysqld_pid>
# 或开 cpuset cgroup 做组绑定
```

**第四步**：如果是 NUMA 系统，还要考虑内存本地性：

```bash
numactl --hardware
numastat -p <pid>
# 看进程的内存都分在哪些节点
```

严重的跨节点访问用 `numactl --cpunodebind=0 --membind=0 ./mysqld` 或 `systemd` 的 `NUMAPolicy=bind` 解决。

## 实战案例 4：长尾调度延迟

**现象**：一个低延迟交易系统 99% 的 request 都 <100us，但 0.1% 会飙到几 ms。

排查路径：

```bash
# 1. 先量化：
sudo runqlat-bpfcc -P 30 1
# -P 按 CPU 分开, 一眼能看到哪颗 CPU 最差

# 2. 看是不是偶发抢占 / 软中断 / timer:
sudo trace-cmd record -e irq:softirq_entry \
                      -e irq:softirq_exit \
                      -e timer:hrtimer_expire_entry \
                      -e sched:sched_switch -- sleep 30
sudo trace-cmd report | grep <your_comm>

# 3. 怀疑 TLB / mm 切换? 看 context_switch 栈:
sudo perf record -g -e sched:sched_switch -p <pid> -- sleep 30
sudo perf report --stdio
```

常见"元凶"：

- 定时器（hrtimer）批量到期；
- RCU callback burst；
- 内存回收 kswapd；
- VM exit / KVM 宿主机其它 guest；
- SMI / BIOS SMM（看 `turbostat` 的 SMI# 列）。

对策：PREEMPT_RT + isolcpus + `nohz_full` + `rcu_nocbs` + 绑中断 + 把内存锁死（`mlockall`）。

## 调优 Checklist（生产环境）

下面这张表是遇到调度/延迟问题时的"通用第一反应"：

| 层 | 可调项 | 常见场景 |
|----|--------|----------|
| 硬件 | 关 C-state / P-state autoselect | 低延迟业务 |
| BIOS | 禁用 SMT、开 NPS | HPC / 低延迟 |
| 内核命令行 | `isolcpus`, `nohz_full`, `rcu_nocbs`, `irqaffinity` | RT / DPDK |
| 编译配置 | PREEMPT / PREEMPT_RT | 硬实时 |
| sysctl | `kernel.sched_*`, `kernel.numa_balancing` | 均衡/延迟权衡 |
| cgroup | `cpu.weight`, `cpu.max`, `cpuset.cpus` | 多租户 |
| 进程 | `taskset`, `chrt`, `nice`, `ionice` | 单个服务 |
| 应用 | mlockall、CPU 亲和、忙等 vs. epoll | 框架级 |

**原则是**：先用最软的工具（`nice`、`cpu.weight`）解决，再考虑 RT / 隔离核这类强手段。任何时候都要有回退方案——`isolcpus` 配置错了容易把整机搞死。

## 本章小结

- 调度延迟的排查遵循"宏观 → 具体 CPU → 具体任务 → 具体事件"的收敛路径；
- PSI、`runqlat`、`perf sched` 是三把最常用的铲子；
- 容器里 90% 的性能异常可以在 `cpu.stat` 里找到线索；
- 真正追到微秒级尾延迟时，往往是中断、软中断、timer、RCU 这些"调度器之外"的角色；
- 调优一定要有指标和基线，别凭感觉改。
