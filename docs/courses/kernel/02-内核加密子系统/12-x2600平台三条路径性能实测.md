# x2600 上如何实测 `AF_ALG`、OpenSSL、`libkcapi` 三条路径的性能与 CPU 占用对比

> 目标:给 x2600 这套 SDK 一份可落地的实测方案，用来比较三条用户态路径:
>
> - 直接 `AF_ALG`
> - OpenSSL
> - `libkcapi`
>
> 重点不是“跑一个数”，而是:
>
> 1. 怎么保证三条路径测的是同一类工作负载
> 2. 怎么把吞吐和 CPU 占用一起看
> 3. 在你这套 x2600 环境里,哪些默认可测,哪些需要先打开包
>
> 配套阅读:

---

## 0. 先给结论

在你当前这套 x2600 SDK 上,做这三条路径对比时要先记住两件事:

1. **默认现状**
   - `OpenSSL` 在 x2600 这类 defconfig 里默认是开的
   - `libkcapi` 默认**没开**
   - `libkcapi` 的测速工具 `kcapi-speed` 默认**没开**
   - `AF_ALG` 这条路径你已经能用(前面已经围绕 `af_alg.c` 和 `algif_*` 写完一套文档了)

2. **当前系统大概率没有硬件 crypto 驱动**
   - 所以 `AF_ALG/libkcapi` 最终很可能落到 `aes-generic` / `sha256-generic`
   - OpenSSL 默认更可能走自己的用户态软件/汇编实现

这意味着这份对比文档真正要回答的是:

> **在“都主要靠 CPU 软件实现”的前提下，三条用户态调用路径各自的吞吐和 CPU 成本差多少。**

---

## 1. 先定义你到底要比什么

不要一上来就把:

- `aes-128-cbc`
- `aes-256-gcm`
- `sha256`
- `rng`

全混到一锅里。

### 推荐分两组测

### 1.1 第一组:最稳妥、最容易做公平对比

- **对称加密**:`AES-128-CBC`
- **摘要**:`SHA-256`

原因:

- `AF_ALG`、OpenSSL、`libkcapi` 三条路径都容易找到对应实现
- 输入输出语义简单
- 结果更容易复现

### 1.2 第二组:可选扩展

- `AES-128-GCM`

原因:

- 更贴近很多真实业务
- 但 AEAD 的输入/输出布局复杂
- 不同工具对测试模式的封装不完全一致

### 1.3 暂时不建议放在第一轮

- `RNG`

因为:

- OpenSSL 的 RNG 语义和内核 `stdrng` / `drbg_*` 不是一个东西
- 很难做完全苹果对苹果的比较

所以这篇文档默认以:

- `AES-128-CBC`
- `SHA-256`

作为第一轮基准。

---

## 2. 这套 SDK 当前默认能跑什么

根据 x2600 SDK 的 buildroot defconfig:

- `BR2_PACKAGE_OPENSSL=y`
- `BR2_PACKAGE_LIBOPENSSL=y`
- `# BR2_PACKAGE_LIBKCAPI is not set`
- `# BR2_PACKAGE_LIBKCAPI_APPS is not set`

这意味着:

### 开箱即用

- OpenSSL 路径
- 直接 `AF_ALG` 路径(前提是内核里 `AF_ALG` 已启)

### 需要先补

如果想测 `libkcapi`，建议至少打开:

```text
BR2_PACKAGE_LIBKCAPI=y
BR2_PACKAGE_LIBKCAPI_SPEED=y
```

如果还想顺带装命令行小工具:

```text
BR2_PACKAGE_LIBKCAPI_DGSTAPP=y
BR2_PACKAGE_LIBKCAPI_ENCAPP=y
BR2_PACKAGE_LIBKCAPI_RNGAPP=y
```

## 3. 测试前的环境统一

如果不先把环境固定住，跑出来的数字意义很有限。

## 3.1 固定 CPU 频率/调频策略

如果目标板支持 cpufreq，尽量切到 `performance`:

```bash
for g in /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor; do
    [ -w "$g" ] && echo performance > "$g"
done
```

如果系统没有 cpufreq，就忽略这步。

## 3.2 绑核

尽量把所有测试绑到同一个 CPU 上,例如 CPU0:

```bash
taskset -c 0 <command>
```

这样吞吐和 CPU 占用更可比。

## 3.3 统一测试时长

推荐:

- 每轮至少 `10s`
- 每个算法做 `3~5` 轮
- 取中位数

## 3.4 统一块大小

推荐至少测 3 档:

- `1024`
- `16384`
- `65536`

原因:

- 1 KiB 看 syscall/接口开销
- 16 KiB 看中等块吞吐
- 64 KiB 看大块稳定吞吐

## 3.5 先预热一次

每个算法正式测前先跑一轮不记成绩,避免:

- 页表/缓存冷启动
- 动态链接首次解析
- OpenSSL provider/engine 首次初始化

---

## 4. 指标怎么记

至少记 4 个指标:

1. **吞吐**: MB/s
2. **CPU 占用**: 单进程 `%CPU`
3. **每 MB 消耗的 CPU 百分比**: `%CPU / MBps`
4. **算法/块大小/线程数**

建议最终整理成这样的表:

| 路径 | 算法 | 块大小 | 线程数 | 吞吐 MB/s | `%CPU` | 备注 |
|---|---|---:|---:|---:|---:|---|
| AF_ALG | AES-128-CBC | 16384 | 1 |  |  | |
| OpenSSL | AES-128-CBC | 16384 | 1 |  |  | |
| libkcapi | AES-128-CBC | 16384 | 1 |  |  | |

---

## 5. CPU 占用怎么测

目标是尽量用 **目标板上大概率能拿到的工具**。

## 5.1 首选: `/usr/bin/time`

如果系统有 GNU time:

```bash
/usr/bin/time -v taskset -c 0 <command>
```

看:

- `Percent of CPU this job got`
- `User time`
- `System time`
- `Elapsed (wall clock) time`

优点:

- 简单
- 不依赖额外包

缺点:

- 某些精简 rootfs 未必带 GNU time

## 5.2 次选: `pidstat`

如果系统装了 `sysstat`:

```bash
pidstat -u -p <PID> 1
```

优点:

- 观察过程中的波动

缺点:

- 嵌入式系统经常没装

## 5.3 最保守: `top`

```bash
top -b -d 1 -n 12 -p <PID>
```

适合:

- 板子很精简
- 只想大致看 `%CPU` 是否接近打满一个核

### 建议

如果可用,优先记录:

- `wall time`
- `Percent of CPU this job got`

这样最容易横向比较。

---

## 6. OpenSSL 路径怎么测

OpenSSL 默认就在这套 defconfig 里开着,所以这条最容易先跑。

## 6.1 AES-128-CBC

```bash
taskset -c 0 openssl speed -elapsed -evp aes-128-cbc -seconds 10 -bytes 16384
```

可换块大小:

```bash
taskset -c 0 openssl speed -elapsed -evp aes-128-cbc -seconds 10 -bytes 1024
taskset -c 0 openssl speed -elapsed -evp aes-128-cbc -seconds 10 -bytes 65536
```

如果想测解密:

```bash
taskset -c 0 openssl speed -elapsed -evp aes-128-cbc -decrypt -seconds 10 -bytes 16384
```

## 6.2 SHA-256

```bash
taskset -c 0 openssl speed -elapsed -evp sha256 -seconds 10 -bytes 16384
```

## 6.3 记录方法

建议配合 `time`:

```bash
/usr/bin/time -v taskset -c 0 openssl speed -elapsed -evp aes-128-cbc -seconds 10 -bytes 16384
```

### 注意

OpenSSL `speed` 测的是 OpenSSL 自己的实现路径。  
除非你额外配置了 engine/provider，否则不要把它误当成“OpenSSL 走 AF_ALG”。

---

## 7. `libkcapi` 路径怎么测

## 7.1 先确认板子上有没有工具

如果已经打开:

```text
BR2_PACKAGE_LIBKCAPI=y
BR2_PACKAGE_LIBKCAPI_SPEED=y
```

目标板上通常会有:

- `kcapi-speed`

先列算法:

```bash
kcapi-speed -l
```

### 为什么先列

因为 `kcapi-speed` 用的算法字符串通常是**展示名**，不是你平时写的 `cbc(aes)` / `sha256` 这种内核简名。  
不同架构/实现下展示名会变化，所以:

> **以 `kcapi-speed -l` 输出为准，复制它实际列出来的名字。**

## 7.2 AES 测法

假设 `kcapi-speed -l` 里能看到类似:

```text
AES(G) CBC(G) 128
```

那就测:

```bash
taskset -c 0 kcapi-speed -c 'AES(G) CBC(G) 128' -b 16384 -t 10
```

块大小同样测:

```bash
taskset -c 0 kcapi-speed -c 'AES(G) CBC(G) 128' -b 1024  -t 10
taskset -c 0 kcapi-speed -c 'AES(G) CBC(G) 128' -b 65536 -t 10
```

## 7.3 SHA-256 测法

先 `kcapi-speed -l | grep -i sha256` 找到实际展示名,然后:

```bash
taskset -c 0 kcapi-speed -c '<这里填 kcapi-speed -l 里实际显示的 SHA256 名字>' -b 16384 -t 10
```

## 7.4 这条路径到底测的是什么

`kcapi-speed` 底层是 `libkcapi`,而 `libkcapi` 底层是 `AF_ALG`。  
所以它测的不是“另一套内核后端”，而是:

> **“AF_ALG + libkcapi 封装后的实际开销”**

这条路径特别适合回答的问题是:

> 直接手写 `AF_ALG` 和用 `libkcapi` 封装相比,有没有明显额外开销?

---

## 8. 直接 `AF_ALG` 路径怎么测

这条路径没有像 OpenSSL `speed` / `kcapi-speed` 那样现成的标准测速工具。  
最直接的办法是:

> 用前面  里的最小示例,扩成一个“固定块大小 + 固定时长循环”的小 benchmark 程序。

## 8.1 推荐测法

### AES-128-CBC

逻辑:

1. `socket/bind/setsockopt/accept`
2. 预先准备:
   - 固定 key
   - 固定 iv
   - 固定长度输入 buffer(如 16 KiB)
3. 循环:
   - `sendmsg(opfd, ...)`
   - `recv(opfd, ...)`
   - 累加处理字节数
4. 到 `10s` 停止
5. `总字节 / 总秒数 = MB/s`

### SHA-256

逻辑:

1. `socket/bind/accept`
2. 固定 16 KiB 输入 buffer
3. 每轮:
   - `send(opfd, buf, len, 0)`
   - `recv(opfd, digest, ds, 0)`
4. 跑 10s

## 8.2 为什么 direct AF_ALG 仍然值得测

虽然 `libkcapi` 也走 `AF_ALG`,但 direct benchmark 仍有意义:

- 能测出“裸 syscall/socket/cmsg”这一层的真实成本
- 能跟 `libkcapi` 比“封装层开销”

## 8.3 一个建议的 AF_ALG 结果字段

建议你的自写 benchmark 输出至少包含:

```text
algo=aes-128-cbc
block=16384
seconds=10.00
bytes=167772160
throughput=16.00 MB/s
```

这样后面做汇总最方便。

## 9. 一套推荐的正式对比矩阵

## 9.1 AES-128-CBC

块大小:

- 1024
- 16384
- 65536

每个块大小测:

- OpenSSL
- libkcapi
- direct AF_ALG

每项:

- 10 秒
- 3 次
- 取中位数

## 9.2 SHA-256

同样测:

- 1024
- 16384
- 65536

因为 hash 对“接口开销 / 大块吞吐”的敏感点和 cipher 不一样,两组一起看更有说服力。

## 11. 结果应该怎么解释

### 11.1 如果 OpenSSL 明显更快

常见原因:

- OpenSSL 自己的软件/汇编实现更成熟
- direct `AF_ALG`/`libkcapi` 有 syscall 往返成本
- 你的系统没有硬件 crypto 驱动,内核端只是 `*-generic`

### 11.2 如果 `AF_ALG` 和 `libkcapi` 很接近

说明:

- `libkcapi` 封装层额外开销不大
- 主要成本还是内核路径本身

### 11.3 如果 direct `AF_ALG` 比 `libkcapi` 快

常见原因:

- `libkcapi` 额外做了抽象/初始化/缓冲管理
- 使用方式没有完全对齐

### 11.4 如果三者都差不多慢

那很可能说明:

- 最终都被 CPU 运算本体主导
- 接口层差异不是瓶颈

## 12. 哪些工具不要拿来做这组三路对比

### 12.1 不建议用 `tcrypt`

你当前源码里有:

- `kernel/kernel-5.10/crypto/tcrypt.c`

但它更适合:

> **测内核内部 Crypto API / driver 本体性能**

不适合拿来和:

- OpenSSL 用户态
- direct `AF_ALG`
- `libkcapi`

做苹果对苹果比较。

原因:

- `tcrypt` 不经过用户态 syscall/socket 路径
- 它测的是“内核内部调用算法”的成本
- 而你这次要比较的是“三种用户态访问路径”

### 12.2 不建议把 `cryptsetup benchmark` 当主结果

它适合粗看系统 cipher 吞吐,但:

- 无法直接拆出 OpenSSL/libkcapi/direct AF_ALG 三条路径
- 中间封装层太多

可以当参考,不要当主表。

## 13. 一个很实用的执行顺序

如果你要真正落地,建议按这个顺序来:

1. **先跑 OpenSSL**
   - 因为默认就有
   - 先拿到一个用户态基线

2. **再跑 direct AF_ALG**
   - 用前面的最小示例扩一版循环 benchmark
   - 看内核 socket 路径大概什么水平

3. **最后补 libkcapi**
   - 打开 `BR2_PACKAGE_LIBKCAPI=y`
   - 打开 `BR2_PACKAGE_LIBKCAPI_SPEED=y`
   - 再看封装层相对 direct AF_ALG 的差距

这样你能最快得到第一轮结论,不用一开始就先改 Buildroot。

---

## 14. 最后一句经验话

在你当前这套 x2600 环境里,最有价值的不是争论“哪条接口更高级”,而是先把这两个问题跑出来:

1. **OpenSSL 的纯用户态路径和内核 `AF_ALG` 路径谁快?**
2. **如果都走软件实现,`AF_ALG` 的 syscall/socket 成本到底有多大?**

这两个答案出来之后,你再决定后面是:

- 继续走 OpenSSL
- 继续走 `AF_ALG/libkcapi`
- 还是去找/补硬件 crypto driver

---

## TL;DR

- 第一轮推荐只测 `AES-128-CBC` 和 `SHA-256`
- 这套 x2600 配置默认有 OpenSSL,默认没有 `libkcapi`
- `libkcapi` 要想方便测速,建议打开:
  - `BR2_PACKAGE_LIBKCAPI=y`
  - `BR2_PACKAGE_LIBKCAPI_SPEED=y`
- OpenSSL 用 `openssl speed -elapsed -evp ...`
- libkcapi 用 `kcapi-speed -l` 先列算法,再按展示名测速
- direct `AF_ALG` 用前面的最小示例扩成固定块大小/固定时长循环 benchmark
- CPU 占用优先配合 `/usr/bin/time -v` 或 `top/pidstat`
- `tcrypt` 不适合拿来做这组三路用户态路径对比
