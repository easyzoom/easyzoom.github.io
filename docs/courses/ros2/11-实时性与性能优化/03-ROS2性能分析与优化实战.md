# ROS2 性能分析与优化实战

## 前言

**C：** 节点越来越多、数据越来越大的系统，难免会遇到"CPU 占用过高"、"消息延迟大"、"内存持续增长"这类性能问题。和普通的程序调试不同，ROS 2 的性能分析需要同时关注节点内部逻辑和节点间通信两个层面。本篇从实用的角度出发，介绍 ROS 2 中常用的性能分析工具和优化策略。

## 性能分析工具链

| 工具 | 用途 | 层面 |
| --- | --- | --- |

## 话题频率与延迟分析

### 测量发布频率

```bash
# 测量话题频率（最近 1000 条的平均值）
ros2 topic hz /scan
# 输出：average rate: 10.123
#    min: 0.080s max: 0.120s std dev: 0.008s window: 1000

# 测量带宽
ros2 topic bw /camera/image_raw
# 输出：average: 24567890 B/s (24.5 MB/s)
```

### 测量端到端延迟

```bash
# 测量从发布到接收的延迟
ros2 topic delay /scan

# 测量特定时间段的延迟
ros2 topic delay /scan --window 100
```

### 数据大小检查

```bash
# 查看消息大小
ros2 topic echo /scan --once | wc -c

# 过大的消息会显著增加延迟
# 激光雷达 360 点：~2KB → 正常
# 激光雷达 1080 点：~8KB → 正常
# 点云：100K 点 → ~4MB → 需要优化
```

## CPU 分析

### 使用 top 观察

```bash
# 查看各节点进程的 CPU 占用
top -p $(pgrep -d',' -f ros2)

# 按线程查看（定位到具体线程）
top -H -p $(pgrep -f my_node)
```

### 使用 perf 深入分析

```bash
# 记录 CPU profile
sudo perf record -g -p $(pgrep -f my_node) -- sleep 10

# 查看报告
sudo perf report

# 常见热点：
# - 序列化/反序列化（消息编解码）
# - TF 查询（lookupTransform）
# - 数学运算（矩阵乘法、点云处理）
```

### Flame Graph

```bash
# 安装 Flame Graph 工具
git clone https://github.com/brendangregg/FlameGraph /opt/FlameGraph

# 生成火焰图
sudo perf script | /opt/FlameGraph/stackcollapse-perf.pl | \
  /opt/FlameGraph/flamegraph.pl > flame.svg

# 在浏览器中打开 flame.svg
```

## 内存分析

### 检测内存泄漏

```bash
# C++：使用 Valgrind
valgrind --leak-check=full ros2 run my_pkg my_node

# 持续监控内存增长
watch -n 1 'ps aux | grep my_node'
```

### Python 内存分析

```bash
# 安装内存分析工具
pip install memray

# 记录内存使用
memray run -o memray.bin ros2 run my_pkg my_node.py

# 生成报告
memray flamegraph memray.bin
```

## 节点间通信优化

### 1. 减少消息体积

```cpp
// 不好：发送全部 360 个点的距离数据
sensor_msgs::msg::LaserScan scan;
scan.ranges.resize(360);  // 可能很多点是无效的

// 好：只发送有效范围内的点
// 或者降低分辨率（360 → 180）
scan.angle_increment *= 2;  // 角分辨率减半
scan.ranges.resize(180);
```

### 2. 使用共享内存传输

```xml
<!-- FastDDS 配置中启用共享内存 -->
<transport_descriptor>
  <transport_id>shm_transport</transport_id>
  <type>SHM</type>
  <segment_size>67108864</segment_size>
</transport_descriptor>
```

### 3. 选择合适的 QoS

```cpp
// 传感器数据：BestEffort + Volatile（允许丢包）
auto sensor_qos = rclcpp::SensorDataQoS();

// 控制命令：Reliable + TransientLocal（不能丢）
auto cmd_qos = rclcpp::ServicesQoS();
```

### 4. 多线程执行器

```cpp
// 默认 SingleThreadedExecutor：所有回调在一个线程
// MultiThreadedExecutor：多个回调并发执行
rclcpp::executors::MultiThreadedExecutor executor;
executor.add_node(node1);
executor.add_node(node2);
executor.spin();
```

### 5. 使用组件化节点（Composition）

将多个节点放在同一进程中，避免进程间通信开销：

```bash
# 将多个节点组合到一个进程中
ros2 run component_container_mt component_container_mt

# 加载组件
ros2 component load /ComponentManager my_pkg my_node_component
ros2 component load /ComponentManager my_pkg another_component
```

组件化的优势：
- 消除进程间通信开销
- 共享内存，无需序列化/反序列化
- 减少 DDS 发现负担

## TF 查询优化

```cpp
// 不好：每次回调都查询 TF（可能阻塞）
void sensor_callback(const SensorMsg::SharedPtr msg) {
    auto transform = tf_buffer_->lookupTransform("base_link", "laser_link",
                                                 tf2::TimePointZero);
    // 处理数据...
}

// 好：使用固定时间戳，带超时
void sensor_callback(const SensorMsg::SharedPtr msg) {
    try {
        auto transform = tf_buffer_->lookupTransform("base_link", "laser_link",
                                                     msg->header.stamp,
                                                     rclcpp::Duration(0, 100000000));
        // 处理数据...
    } catch (tf2::TransformException &ex) {
        RCLCPP_WARN(logger, "TF lookup failed: %s", ex.what());
        return;
    }
}
```

## Python 节点性能优化

### 使用 NumPy 加速

```python
# 不好：纯 Python 循环处理点云
for point in points:
    transformed.append(transform_point(point))

# 好：NumPy 向量化操作
import numpy as np
points_array = np.array(points)
transformed = rotation_matrix @ points_array.T + translation
```

### 避免频繁创建对象

```python
# 不好：每次回调都创建新消息
def callback(self, msg):
    result = PointCloud2()
    # ... 填充 result ...

# 好：预分配对象
def __init__(self):
    self._result = PointCloud2()

def callback(self, msg):
    self._result.header = msg.header
    # ... 复用 self._result ...
```

### 使用 asyncio 处理 I/O

```python
import asyncio

class MyAsyncNode(Node):
    def __init__(self):
        super().__init__('async_node')
        self.loop = asyncio.new_event_loop()

    def timer_callback(self):
        # 非阻塞 I/O
        asyncio.run_coroutine_threadsafe(
            self.async_io_operation(), self.loop)
```

## 性能优化检查清单

| 检查项 | 工具 | 目标值 |
| --- | --- | --- |
| CPU 占用率 | `top` | < 80%（留裕度） |
| 话题频率 | `ros2 topic hz` | 达到预期频率 |
| 端到端延迟 | `ros2 topic delay` | < 控制周期的 50% |
| 内存增长 | `watch ps aux` | 无持续增长 |
| DDS 发现时间 | 观察节点 list | < 5s |
| TF 查询耗时 | perf | < 1ms |

## 常见问题

### 节点 CPU 占用 100%

1. **死循环**：回调中没有 `rate.sleep()` 或 `spin_some` 参数不对
2. **TF 查询阻塞**：`lookupTransform` 在循环中无超时等待
3. **序列化开销**：大消息的高频发布
4. **日志刷屏**：高频日志输出

### 消息延迟突然增大

1. 检查是否有新节点加入导致 DDS 重新发现
2. 检查网络状况（`ping`、`iperf`）
3. 检查接收端是否有阻塞操作
4. 检查系统负载（`uptime`）

### Python 节点特别慢

1. 使用 `MultiThreadedExecutor`
2. 将计算密集型代码用 Cython 或 C++ 重写
3. 使用 NumPy 向量化代替 Python 循环
4. 减少不必要的消息复制

## 小结

ROS 2 性能优化是系统性的工作：

1. **测量先行**：先测量再优化，不要盲目猜测瓶颈
2. **话题分析**：`hz`、`delay`、`bw` 是第一手数据
3. **CPU 分析**：`top` 定位热点，`perf` 深入分析
4. **通信优化**：QoS 选择、共享内存、组件化
5. **内存分析**：`valgrind`（C++）、`memray`（Python）
6. **TF 优化**：带超时的查询、缓存变换结果
7. **Python 优化**：NumPy 向量化、避免频繁 GC
