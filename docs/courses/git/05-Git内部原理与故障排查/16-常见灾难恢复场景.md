# 常见灾难恢复场景

## 前言

**C：** 用 Git 难免会遇到"完了完了"的时刻——代码没了、分支没了、历史乱了。好消息是，Git 的设计理念就是"几乎不会丢失数据"。本文汇总最常见的灾难场景和对应的恢复方法，建议收藏备用。

<!-- more -->

## 场景一：误删了本地分支

```shell
# 不小心删除了分支
git branch -D important-feature
```

**恢复方法：**

```shell
# 1. 用 reflog 找到分支最后的提交
git reflog
# 找到类似这样的记录：
# e5f6g7h HEAD@{3}: checkout: moving from important-feature to main

# 2. 重新创建分支
git switch -c important-feature e5f6g7h
```

::: tip 笔者说
如果你还记得分支名，可以直接用 `git reflog show <branch-name>` 查看该分支的操作历史。
:::

## 场景二：reset --hard 丢失了提交

```shell
# 你有 3 个未推送的新提交
git reset --hard HEAD~3
# 现在这 3 个提交"不见了"
```

**恢复方法：**

```shell
# 1. 查看 reflog
git reflog
# a1b2c3d HEAD@{0}: reset: moving to HEAD~3  ← 刚才的 reset
# e5f6g7h HEAD@{1}: commit: third commit     ← 丢失的第一个提交
# d4e5f6g HEAD@{2}: commit: second commit
# b3c4d5e HEAD@{3}: commit: first commit

# 2. 恢复
git reset --hard HEAD@{1}
# 或
git reset --hard e5f6g7h
```

### 如果只是想恢复某些文件

```shell
# 不想全部恢复，只要某个文件
git checkout HEAD@{1} -- path/to/file.js

# 或使用 restore
git restore --source=HEAD@{1} path/to/file.js
```

## 场景三：commit 时漏掉了文件

```shell
# 提交后发现有文件漏掉了
git commit -m "feat: add login"
# 但忘了 git add utils.js
```

**恢复方法：**

```shell
# 方法一：追加提交（推荐，不改写历史）
git add utils.js
git commit -m "feat: add login utils"

# 方法二：修正上一个提交（仅限未推送）
git add utils.js
git commit --amend --no-edit
```

## 场景四：写错了提交信息

```shell
# 提交信息写错了
git commit -m "feat: add loign"  # 拼写错误
```

**恢复方法：**

```shell
# 未推送：直接修改
git commit --amend -m "feat: add login"

# 已推送：创建新的修正提交（不改写历史）
git commit --allow-empty -m "docs: fix typo in previous commit message"

# 或者交互式 rebase（谨慎！需要 force push）
git rebase -i HEAD~1
# 将 pick 改为 reword
```

## 场景五：不小心把敏感信息提交了

```shell
# 把密码或密钥提交到了仓库
git commit -m "add config"
# .env 中包含了数据库密码
```

**恢复方法：**

```shell
# 第一步：从 Git 历史中彻底删除
# 使用 git filter-branch（旧方法）
git filter-branch --force --index-filter \
  'git rm --cached --ignore-unmatch .env' \
  --prune-empty --tag-name-filter cat -- --all

# 或使用 git filter-repo（新方法，更高效）
pip install git-filter-repo
git filter-repo --path .env --invert-paths

# 第二步：强制推送（通知所有团队成员）
git push --force --all
git push --force --tags

# 第三步：轮换密钥！
# 即使从历史中删除了，之前的密钥可能已泄露
# 必须更换密码/密钥/Token
```

::: warning 严重警告
敏感信息提交后，即使从历史中删除，也要假设已经泄露。务必更换相关的密码、密钥和 Token。
:::

## 场景六：merge 搞乱了历史

```shell
# 不小心合并了错误的分支
git merge wrong-branch
# 或者合并时产生了混乱的冲突解决
```

**恢复方法：**

```shell
# 方法一：reset（未推送时）
git reset --hard HEAD@{1}

# 方法二：revert（已推送时）
git revert -m 1 HEAD
```

## 场景七：rebase 导致冲突不断

```shell
# rebase 到一半，冲突太多想放弃
git rebase main
# 几十个冲突，已经处理不过来了
```

**恢复方法：**

```shell
# 随时可以放弃 rebase
git rebase --abort

# 这会回到 rebase 之前的状态
```

### rebase 完成后发现结果不对

```shell
# rebase 完成了但结果不对
git reflog
# 找到 rebase 之前的记录
git reset --hard HEAD@{n}
```

## 场景八：force push 覆盖了队友的提交

```shell
# 你 force push 覆盖了远程分支
git push --force origin feature
# 队友的提交丢失了
```

**恢复方法：**

```shell
# 队友侧操作：
# 1. 查看本地 reflog
git reflog show feature
# e5f6g7h feature@{0}: commit: my work  ← 队友的提交
# a1b2c3d feature@{1}: commit: shared work

# 2. 基于队友的提交创建恢复分支
git switch -c feature-recover e5f6g7h

# 3. 通知你（force push 的人）合并队友的提交
git push -u origin feature-recover
# 然后创建 PR 合并回 feature
```

## 场景九：.git 目录被误删/损坏

```shell
# .git 目录被删除
rm -rf .git

# 或者 .git 目录损坏
```

**恢复方法：**

```shell
# 如果有远程仓库：
git clone <remote-url> .
# 注意：这会覆盖本地文件！如果本地有未推送的修改，先备份

# 如果没有远程仓库但有备份：
# .git 目录被删除后，本地的文件内容还在
# 但所有历史、分支、提交信息都丢失了
# 只能重新 init 并提交

git init
git add .
git commit -m "recovered from .git deletion"
```

::: warning 注意
`.git` 目录损坏是比较严重的灾难。如果没有远程仓库的备份，历史记录无法恢复。定期推送到远程是最好的防护。
:::

## 预防措施

### 1. 定期推送

```shell
# 养成习惯：每天下班前 push
git push
```

### 2. 重要操作前创建备份分支

```shell
# rebase/reset 之前创建安全点
git branch backup-before-rebase

# 操作完成后可以删除
git branch -D backup-before-rebase
```

### 3. 使用 pre-commit hook 检查敏感信息

```shell
# 安装检测工具
npm install -g git-secrets

# 配置检查规则
git secrets --register-aws
git secrets --install
```

### 4. 配置自动备份

```shell
# 使用 cron 定期推送所有分支
# crontab -e
# 0 * * * * cd /path/to/repo && git push --all origin
```

## 应急流程速查表

| 灾难 | 恢复命令 |
|------|---------|
| 误删分支 | `git reflog` → `git switch -c <branch> <hash>` |
| reset --hard | `git reflog` → `git reset --hard HEAD@{n}` |
| 错误合并 | `git reset --hard HEAD@{1}` 或 `git revert -m 1 HEAD` |
| rebase 失败 | `git rebase --abort` |
| 丢失提交 | `git reflog` → `git switch -c <name> <hash>` |
| 漏提交文件 | `git add` → `git commit --amend` |
| 敏感信息泄露 | `git filter-repo` → 轮换密钥 |
| stash drop | `git fsck --dangling` → `git stash apply <hash>` |

## 小结

- Git 几乎不会真正丢失数据，关键是知道怎么找回来
- **reflog** 是第一恢复工具，90% 的场景都能用它解决
- 重要操作前创建备份分支是最简单的预防措施
- 敏感信息泄露后必须轮换密钥
- 定期推送是最后的保险
