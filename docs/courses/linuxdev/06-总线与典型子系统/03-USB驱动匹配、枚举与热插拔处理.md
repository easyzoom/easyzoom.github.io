# USB驱动匹配、枚举与热插拔处理

## 前言

**C：** USB 驱动和很多板级驱动最大的不同，不是总线名字不同，而是它天然就是动态世界：设备会插上、拔掉、重枚举、挂起、恢复，还会有不同接口、不同 endpoint、不同速率等级。很多工程师把 USB 驱动写成“设备一旦 probe 成功就永远在线”，结果热插拔和异常恢复一来，整个结构就垮了。本篇重点讲 USB 驱动为什么一定要从生命周期和状态机视角来设计。

## USB 驱动的动态生命周期

```mermaid
flowchart LR
  plugIn["设备插入"] --> enumerate["总线枚举与描述符解析"]
  enumerate --> match["驱动匹配到接口"]
  match --> probe["probe建立状态"]
  probe --> runState["正常收发与事件处理"]
  runState --> disconnect["拔出或断开"]
  disconnect --> cleanup["资源回收与状态终止"]
```

## USB 匹配的基本单位常常不是“整个设备”

很多 USB 驱动的匹配对象并不是整台设备，而是：

- 某个 interface
- 某组 endpoint 能力

这和很多平台设备、字符设备的思路不同。  
高级工程师必须先看清：

- 设备暴露了几个 interface
- 哪个 interface 才是驱动负责的对象
- 控制、数据、事件 endpoint 如何分工

如果这层边界不清，后面容易把状态绑错位置。

## probe 的关键不是“建对象”，而是“建生命周期边界”

USB probe 里要做的当然包括分配状态、解析 endpoint、申请 URB、注册接口，但真正重要的是：  
**你要知道这些对象在 disconnect 时如何安全退出。**

因为 USB 最大的现实约束是：

- 设备可能随时消失
- 正在飞行中的 URB 可能还没完成
- 用户态可能还持有 fd
- workqueue / 回调可能还在跑

这意味着 USB 驱动比很多静态设备更强调：

- 引用计数
- 取消与回收顺序
- 状态机终止条件

## 热插拔为什么特别考验结构设计

热插拔场景最容易暴露这些设计缺陷：

1. 回调还在用已释放对象
2. disconnect 只释放了一半资源
3. 用户态还在读写，但设备已经不在
4. 多个异步路径没有统一“设备已离线”语义

所以成熟 USB 驱动通常会尽早建立一个核心状态位，例如：

- online / offline
- present / removed
- running / shutting_down

再围绕这个状态组织：

- 新请求是否还允许进入
- 已提交 URB 如何终止
- 资源何时真正释放

## URB 视角下的驱动思维

USB 驱动很大一部分复杂度都落在 URB 生命周期：

- 提交
- 完成回调
- 取消
- 重提交流

所以排障时，你不能只看“有没有发包/收包”，还要看：

- URB 是否真的提交成功
- 完成回调是否总能收敛
- disconnect 时是否把悬挂 URB 全部处理干净

## suspend / resume 与热插拔经常交织

USB 设备除了插拔，还会遇到：

- autosuspend
- 系统 suspend
- 恢复后重新协商状态

因此 USB 驱动不能把 PM 视为“另一个独立话题”，而要和热插拔一起考虑。  
很多疑难 bug 就发生在：

- 恢复过程中设备状态半初始化
- 用户请求提前进来
- 旧 URB、新配置和离线状态混在一起

## 一些常见反模式

1. 把 USB 驱动写成静态设备驱动  
   忽略设备随时可能断开。
2. disconnect 只做表面释放  
   没有真正收敛异步路径。
3. 没有清晰的 offline 状态语义  
   导致新旧请求交错。
4. 只在正常插拔路径测试  
   不测高频插拔、异常拔出、挂起恢复。

## 一句经验总结

USB 驱动最核心的不是“识别了哪个 VID/PID”，而是你是否把动态生命周期设计对了。  
枚举、匹配、收发都只是过程；真正决定驱动稳定性的，是热插拔和异步回调能否安全收敛。
