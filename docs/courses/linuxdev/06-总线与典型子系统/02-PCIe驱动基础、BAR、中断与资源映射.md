# PCIe驱动基础、BAR、中断与资源映射

## 前言

**C：** 如果说平台驱动更像“板上固定设备”，那 PCIe 驱动更像“被枚举出来的高速可插拔设备”。很多工程师第一次写 PCIe 驱动时，会被 BAR、配置空间、MSI/MSI-X、中断向量、DMA 能力协商这些概念一下子压住。其实它的核心脉络很清楚：先枚举、再资源、后中断、最后数据通路。本篇先建立一套足够稳的 PCIe 驱动心智模型。

## PCIe 驱动初始化主线

```mermaid
flowchart LR
  enumDev["PCI枚举发现设备"] --> enableDev["pci_enable_device"]
  enableDev --> requestBars["申请并映射BAR"]
  requestBars --> setupDma["配置DMA能力"]
  setupDma --> setupIrq["申请MSI或MSI-X"]
  setupIrq --> bringup["初始化队列与数据通路"]
```

## PCIe 驱动和平台驱动最大的不同

平台驱动更依赖：

- 设备树
- 固定资源描述
- SoC 板级集成

PCIe 驱动更依赖：

- 总线枚举
- Vendor ID / Device ID 匹配
- 配置空间能力协商

也就是说，PCIe 驱动不是“我假设这个设备在那儿”，而是“总线告诉我它出现了，而且资源已经被发现出来了”。

## BAR 到底是什么

BAR（Base Address Register）本质上是设备向系统声明的一段资源窗口。  
它可能表示：

- MMIO 寄存器空间
- 设备本地内存窗口
- 某些历史兼容 I/O 空间

驱动最常做的是：

1. 获取 BAR 资源信息
2. 申请资源所有权
3. 把 MMIO 映射到 CPU 可访问地址

要注意的是：  
BAR 映射出来的是 CPU 访问设备寄存器/窗口的入口，不是 DMA 缓冲区。

## BAR 和 DMA 是两条不同链路

这是 PCIe 新手最容易混淆的点之一：

- BAR：CPU 访问设备
- DMA：设备访问内存

一个成熟 PCIe 驱动通常同时处理这两条链：

- 通过 BAR 操作控制寄存器、doorbell、状态位
- 通过 DMA 队列、描述符和中断完成数据搬运

如果这两条链没分清，后面调试会非常混乱。

## 中断为什么优先考虑 MSI/MSI-X

现代 PCIe 驱动里，更常见的是：

- MSI
- MSI-X

因为它们相较传统 INTx 更适合：

- 多队列
- 高并发
- 降低共享中断复杂度

MSI-X 尤其适合：

- 每队列独立中断向量
- CPU 亲和优化
- 高吞吐设备的扩展设计

所以高级工程师在设计 PCIe 驱动时，通常会尽早思考：

- 设备有几个队列
- 每个队列是否独立中断
- 中断与 CPU 亲和性如何规划

## DMA 能力协商不能草率

PCIe 设备经常需要 64-bit DMA、IOMMU 协同、较大描述符环。  
所以驱动在 bring-up 早期就要确认：

- DMA mask 是否设置成功
- 是否支持期望地址宽度
- IOMMU 打开后的映射是否正常

这里如果只在某台开发机上“碰巧可用”，量产时会经常暴露兼容性问题。

## 一个成熟的 probe 关注什么

成熟的 PCIe probe 一般会非常强调失败清理路径，因为初始化步骤多、资源链长。  
典型顺序包括：

1. 使能设备
2. 请求和映射 BAR
3. 配置 DMA mask
4. 建立软件队列与描述符
5. 申请 MSI/MSI-X
6. 注册上层子系统接口

任何一步失败，都要确保前面的资源能对称回滚。  
这也是 PCIe 驱动里最容易被低估的工程能力。

## 常见问题模式

1. BAR 映射成功但寄存器访问全错  
   先看设备文档中的寄存器窗口和 endian 假设。
2. 中断申请成功但没有完成事件  
   先分清设备是否真的发了中断，还是软件没使能。
3. DMA 偶发失败  
   先排 DMA mask、IOMMU、同步方向和描述符生命周期。
4. 热插拔或错误恢复不稳定  
   这通常不是单个 API 问题，而是整体状态机没设计好。

## 一句经验总结

PCIe 驱动的主线很明确：枚举、BAR、DMA、中断、队列。  
但真正的难点不在“知道这些名词”，而在于把资源链路、错误清理和多队列中断设计成可长期维护的结构。
