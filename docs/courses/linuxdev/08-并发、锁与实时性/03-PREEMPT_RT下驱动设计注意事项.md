# PREEMPT_RT下驱动设计注意事项

## 前言

**C：** 很多驱动在普通内核上“问题不大”，一切换到 PREEMPT_RT，就会暴露出延迟抖动、锁语义误判、线程优先级冲突、IRQ 处理路径设计不当等问题。原因并不是 RT 内核“太严格”，而是它把你平时侥幸没撞到的设计边界放大了。高级驱动工程师如果只会在通用发行版内核上验证，往往很难支撑工业控制、音视频低时延、机器人等场景。本篇专门从驱动设计视角讲 RT 约束。

## RT 内核下的关注点

```mermaid
flowchart LR
  irqThread["IRQ线程化"] --> lockSemantics["锁语义变化"]
  lockSemantics --> latencyBudget["延迟预算更敏感"]
  latencyBudget --> priorityControl["优先级与抢占控制"]
  priorityControl --> driverDesign["驱动设计需要重新审视"]
```

## 不要把 PREEMPT_RT 理解成“只是更快”

PREEMPT_RT 更准确的目标是：

- 降低最坏情况延迟
- 提高抢占能力
- 让系统行为更可预测

它关注的是 tail latency，而不是单纯平均吞吐。  
因此很多驱动在 RT 环境里要重新审视：

- 锁持有时间
- IRQ 处理结构
- 工作线程优先级
- 共享资源争用

## 自旋锁语义在 RT 下并不是原样不动

RT 环境里，很多“你以为会一直忙等”的路径，底层语义可能发生变化。  
这意味着：

- 不能把普通自旋锁当作“永远极短、完全不可睡眠”的简单假设
- 真正必须保持原子语义的地方，要更慎重地区分 `raw_spinlock_t`

但高级工程师要记住一点：  
不是所有地方都该上 `raw_spinlock_t`。  
它的使用应该非常克制，只留给真正需要保留最原始中断/调度语义的关键区。

## IRQ 线程化会改变你的处理习惯

在 RT 系统中，中断线程化往往更常见。  
这带来的直接影响是：

- 一部分原本“默认硬中断就地做”的工作，调度特征变了
- IRQ 处理与普通线程调度、优先级继承的关系变得更重要

因此驱动代码要尽量避免：

- 中断线程里做不可控长耗时路径
- 中断线程与普通工作线程抢同一个大锁
- 没有明确优先级边界

## 最大的敌人通常不是平均性能，而是长尾

RT 场景里，系统出问题往往不是“整体慢了 10%”，而是：

- 偶发 10ms、50ms、100ms 尾延迟
- 某个锁竞争把关键线程卡住
- 中断线程优先级不当导致关键事件迟到

所以你在 RT 下看驱动，应该更关注：

- 哪个路径持锁时间最长
- 哪个路径最容易发生优先级反转
- 哪个线程真正决定设备响应上界

## 优先级反转为什么在驱动里更危险

如果高优先级任务等待一个被低优先级线程持有的锁，而中优先级任务又不断抢占低优先级线程，就会发生优先级反转。  
在用户态这已经很麻烦，在驱动里更麻烦，因为它常常叠加：

- IRQ 线程
- workqueue
- 用户态控制线程
- DMA 完成回调

一旦这些路径共享锁或状态对象，延迟抖动就会非常难看。

## RT 下更推荐的驱动结构

一个更稳健的结构通常是：

1. 缩小硬实时路径  
   只保留真正必须立即完成的动作
2. 限制共享状态边界  
   不让所有线程争同一把大锁
3. 拆分快慢路径  
   把恢复、日志、统计等慢工作挪走
4. 明确优先级  
   知道谁在影响最坏情况延迟

## 一些容易忽略的细节

### 日志过多

在 RT 场景里，热路径里频繁日志本身就可能成为扰动源。

### 大块内存分配

动态分配、回收、内存碎片引发的额外开销，在 RT 下更容易变成尾延迟来源。

### 不必要的全局锁

很多普通内核下“还能接受”的大锁，在 RT 下会快速成为瓶颈。

## 排查 RT 问题的建议顺序

1. 先看关键 IRQ / 线程的调度与优先级
2. 再看最长持锁路径
3. 再看中断线程和 workqueue 的职责是否混乱
4. 最后再看驱动是否存在本不该进入热路径的慢操作

这套顺序很有用，因为 RT 问题常常不是单点 bug，而是结构性问题。

## 一句经验总结

普通内核上的“基本正确”，不等于 RT 场景下的“可交付”。  
PREEMPT_RT 会逼你面对真正的延迟预算、锁边界和优先级设计，这也是高级驱动工程师必须具备的能力之一。
