# ftrace、tracepoints与函数级时序分析

## 前言

**C：** 高级驱动排障最怕两件事：一是日志太少，根本不知道发生了什么；二是日志太多，却仍然看不出时序关系。`printk` 适合做存在性确认，但一旦问题变成“哪个函数先跑、哪段路径拖慢、谁打断了谁”，就要进入 trace 世界。`ftrace` 和 tracepoints 的价值，不在于工具名字多酷，而在于它们能把驱动问题从“猜测”变成“时序证据”。

<!-- more -->

## trace 视角下的问题定位链路

```mermaid
flowchart LR
  symptom["发现超时/抖动/卡顿"] --> chooseTrace["选择trace点"]
  chooseTrace --> capture["采集函数或事件时序"]
  capture --> correlate["关联IRQ/调度/驱动路径"]
  correlate --> isolate["锁定慢函数或异常顺序"]
```

## 为什么 `printk` 不够

`printk` 的确非常有用，但在高级问题里经常不够：

- 日志会扰动时序
- 高频路径里不敢多打
- 很难准确看出函数嵌套与耗时
- 线程调度和 IRQ 时序关联不直观

所以当问题升级为：

- 偶发超时
- 某条中断链路变慢
- 某个函数偶尔执行异常久

你就该优先考虑 trace，而不是一味补更多日志。

## `ftrace` 最擅长什么

`ftrace` 特别适合回答这些问题：

- 哪些函数真的被调用了
- 调用顺序是什么
- 哪个函数耗时最长
- 是谁在打断当前路径

常见模式包括：

- function trace
- function graph trace
- 调度相关事件
- IRQ 相关事件

对于驱动工程师来说，最常用的是：  
**把驱动函数、IRQ 事件、调度事件放到同一时间线上看。**

## tracepoints 为什么重要

tracepoints 的价值在于：

- 开销和语义通常比大面积函数跟踪更可控
- 可以稳定表达“某类事件发生了”
- 更适合长期保留为工程化观测点

例如你可以关注：

- 中断进入/退出
- 调度切换
- workqueue 执行
- 块层、网络层、PM 事件

这类信息通常比“把整个内核所有函数都追一遍”更有诊断价值。

## 一个实用的分析套路

假设你遇到“设备偶发超时”，推荐用下面这套顺序：

1. 确认超时发生的上层路径  
   是提交命令超时、DMA 完成超时，还是 resume 超时
2. 选最关键的 3 类事件  
   驱动函数、IRQ 事件、调度切换
3. 做短时间窗口采集  
   不要一上来就开极宽范围
4. 找“最后一个正常事件”和“第一个异常间隙”  
   这是最快缩小范围的方法

高级工程师不是“把 trace 开满”，而是懂得缩小观察窗口。

## function graph 特别适合找什么

如果你怀疑某个回调链过长，function graph 往往很好用。  
它能帮助你看出：

- 调用栈层次
- 哪一层最耗时
- 是否发生了异常重入或递归

这对于下面几类问题尤其有价值：

- `probe()` / `resume()` 为什么慢
- 某个 `ioctl` 路径为什么偶发卡住
- 中断线程为什么偶发执行很久

## 什么时候该自定义 tracepoints

如果一个驱动需要长期维护，且常常出现“线上偶发、实验室难复现”的问题，可以考虑在关键状态切换点加自定义 tracepoints。  
比起零散 `pr_info`，它更适合长期做：

- 状态迁移观测
- 性能时延统计
- 现场采样

这也是高级工程化意识的一部分：  
为未来的排障预留可观测性。

## 常见误区

1. 问题一复杂就全靠 `printk`
2. 一上来就 trace 全内核，结果数据淹没人
3. 只看函数调用，不看调度和 IRQ 事件
4. 采集时间过长，最后自己把系统扰动得更严重

## 一句经验总结

高级驱动排障的关键，不是“工具越多越好”，而是**把证据按时间线排出来**。  
只要你能把驱动函数、IRQ、调度和状态切换对齐到同一条时间线上，大多数“偶发问题”都会比只看日志清楚得多。
