# IOMMU、ScatterGather与高性能数据通路设计

## 前言

**C：** 写到 DMA 进阶阶段，真正拉开水平差距的，已经不是“会不会用 `dma_map_single()`”，而是能不能把整条数据通路设计得既正确又高效。为什么某些驱动一上大包、大吞吐、多队列就抖动？为什么打开 IOMMU 后性能掉了、但安全性和隔离性上去了？为什么同样是链表描述符，有人能把吞吐跑满，有人却卡在映射和同步上？本篇就把 IOMMU、Scatter-Gather 和高性能数据路径放在一起讲。

```mermaid

  userBuffer["页或缓冲片段"] --> sgTable["scatterlist"]
  sgTable --> iommuMap["dma_map_sg"]

  dmaSegments --> hwDesc["硬件描述符链"]
  hwDesc --> deviceEngine["DMA引擎"]
```

## 为什么需要 IOMMU

IOMMU 的价值不止一个“地址翻译”，更重要的是：

- 给设备提供独立的 I/O 地址空间
- 限制设备只能访问被映射的内存区域
- 为复杂系统中的隔离、安全和虚拟化打基础

从驱动视角看，IOMMU 带来的变化是：

- 设备看到的地址不再等于物理地址
- map/unmap 成本可能更明显
- 错误会以 fault 形式更早暴露

所以高级工程师不能再用“物理地址直通”的心智模型来写所有平台。

## Scatter-Gather 解决什么问题

真实系统里，大块数据经常不是一整段连续物理内存。  
例如网络包、文件页缓存、视频帧缓冲，都可能由多个片段组成。

如果每次都强行拷贝成连续缓冲区，会带来：

- 额外 CPU 开销
- 更高延迟
- 缓冲复制放大

Scatter-Gather 的思路是：

- 保留数据本来的分散布局
- 通过 `scatterlist` 描述各段
- 让 DMA 引擎按段搬运

这就是为什么高性能驱动普遍依赖 SG，而不是一味追求“大连续缓冲”。

## `dma_map_sg()` 的工程意义

对驱动来说，`dma_map_sg()` 不只是一个 API，它代表着：

- 把 CPU 视角的一组分散片段
- 转换成设备可访问的一组 DMA 段

注意这里是“一组 DMA 段”，不是机械地“一项进一项出”。  
经过 IOMMU 或 DMA 层处理后：

- 段数可能被合并
- 返回值可能小于原始 scatterlist 项数

所以后续编程时要以返回后的可映射段数为准，而不是想当然用原列表长度。

## 高性能路径最怕什么

### 频繁 map/unmap

如果每个小包、每次小块搬运都做完整 map/unmap，IOMMU 和 DMA 层开销会被放大。  
高负载下，问题常表现为：

- 吞吐上不去
- CPU 占用偏高
- 尾延迟飙升

### 描述符组织不合理

即使 DMA 引擎很强，如果描述符链太碎、门铃频率太高、批量提交太少，性能仍然会浪费在控制路径上。

### 同步粒度过细

每次只同步极小区间、频繁在 CPU/设备间来回切所有权，会让 cache 和内存子系统承受不必要压力。

## 高级驱动常用的优化方向

1. **缓冲池复用**  
   减少临时分配与释放。
2. **批量提交**  
   一次下发多项描述符，降低 doorbell/寄存器开销。
3. **环形队列设计**  
   让生产者和消费者边界清晰，减少锁竞争。
4. **长生命周期映射**  
   对某些固定缓冲池，避免重复映射成本。
5. **队列与 CPU 亲和性规划**  
   让中断、NAPI、工作线程和 DMA 队列尽量在局部拓扑内协同。

## IOMMU 打开后性能下降，先看什么

很多团队一看到性能下降，就想把 IOMMU 关掉。  
但高级工程师更应该先判断：

- 是否 map/unmap 过于频繁
- 是否用了过碎的 SG 片段
- 是否描述符批量过小
- 是否缺少缓冲池复用
- 是否 fault/flush 异常导致额外开销

IOMMU 的确可能带来成本，但很多下降不是 IOMMU“本身太慢”，而是原来的驱动路径过度依赖物理直通假设。

## 一个更成熟的数据通路视角

设计高性能驱动时，要把链路分成三层来想：

### 数据层

数据本体怎么布局？是页片段、线性缓冲，还是混合模型？

### 映射层

哪些对象需要长期映射，哪些只做短期映射？同步和解绑边界在哪？

### 提交层

设备描述符、队列深度、中断合并、批量提交策略是否匹配硬件能力？

只盯单个 API，永远优化不到位；必须从这三层一起看。

## 一些常见误区

1. SG 一定比连续缓冲慢。  
   不一定，很多高吞吐设备就是为 SG 设计的。
2. 打开 IOMMU 就意味着性能必然不可接受。  
   这取决于工作负载和驱动组织方式。
3. 返回的 SG 段数一定等于原始项数。  
   这是很常见的错误假设。
4. 高性能优化只关心 DMA。  
   实际上锁、中断、NUMA、cache 拓扑都会一起影响结果。

## 对高级工程师的要求

真正成熟的驱动工程师，需要能同时回答三类问题：

- 正确性：设备是否一定访问到合法且同步好的内存
- 性能：这条路径是否把 CPU 和 IOMMU 开销压到了合理范围
- 可维护性：换平台、开安全特性、增大并发后是否还能稳定工作

这也是为什么 DMA 进阶内容，最终一定会和并发、实时性、trace、性能分析连起来。
