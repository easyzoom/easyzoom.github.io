# suspend与resume流程及常见陷阱

## 前言

**C：** 系统级 suspend/resume 和 runtime PM 看起来都叫“电源管理”，但它们对驱动的挑战并不一样。runtime PM 更像设备忙闲切换，系统 suspend/resume 则像一次全系统级的状态冻结、断电与恢复。很多驱动平时运行完全正常，一到休眠唤醒就出各种诡异问题：设备失联、寄存器默认值丢失、中断不再触发、DMA 队列死掉。本篇就讲为什么 suspend/resume 是量产驱动的硬门槛。

## 系统休眠恢复主线

```mermaid
flowchart LR
  running["系统运行中"] --> suspendPrep["准备suspend"]
  suspendPrep --> quiesce["停止新请求并收敛在途操作"]
  quiesce --> powerLoss["进入低功耗或断电态"]
  powerLoss --> resumeRestore["resume恢复时钟/电源/上下文"]
  resumeRestore --> reenable["重新使能中断/队列/接口"]
  reenable --> running
```

## suspend/resume 最容易被低估的点

驱动开发里最危险的误解之一是：  
“resume 不就是把之前的初始化再来一遍吗？”

真实情况通常复杂得多，因为系统 suspend 会带来：

- 设备上下文可能丢失
- 父设备/时钟/电源域顺序变化
- 中断线状态变化
- DMA/队列/飞行中事务必须先收敛

所以 resume 不是“重新 probe”，而是“在已有对象和上层关系仍然存在的前提下，把设备状态恢复到可继续运行”。

## suspend 前最重要的是收敛在途操作

一个成熟的 suspend 入口通常先做这些事：

- 阻止新请求进入
- 等待正在处理的事务收敛
- 停止 DMA/队列
- 屏蔽或妥善处理中断

如果这一步没做干净，后面常见问题包括：

- 设备在断电前仍有事务进行
- 唤醒后软件状态与硬件状态错位
- resume 后第一笔请求直接超时

## resume 的关键不是“开电”，而是“恢复顺序”

恢复顺序经常决定驱动是否稳定。  
常见顺序大致包括：

1. 恢复供电和时钟
2. 恢复寄存器或上下文
3. 重新建立 DMA/队列
4. 清中断状态
5. 重新开放请求入口

如果顺序错了，可能出现：

- 队列先开放，但寄存器还没恢复
- 中断先开了，但设备状态仍半初始化
- 上层开始发请求，但底层资源还没就绪

## 最常见的 4 类陷阱

### 陷阱一：只测轻度 suspend

很多平台上浅睡和深睡的硬件行为完全不同。  
浅睡能过，不代表深睡一定没问题。

### 陷阱二：认为寄存器值一定保留

有些设备或电源域恢复后寄存器会回默认值，必须显式重写配置。

### 陷阱三：忘了和 runtime PM 协同

系统 PM 和 runtime PM 混用时，如果状态边界不清，很容易造成双重 suspend、重复 resume 或引用计数混乱。

### 陷阱四：只测“睡一次醒一次”

真正产品里更容易出问题的是：

- 连续多次 suspend/resume
- 高负载下进入 suspend
- 刚恢复就立刻高并发访问

## 排查 suspend/resume 问题的顺序

1. 先确认 suspend 前是否真的 quiesce 成功
2. 再确认 resume 后寄存器/时钟/电源是否恢复完整
3. 再看中断和 DMA 路径是否重新联通
4. 最后检查上层是否在设备完全 ready 前就开始访问

这套顺序非常关键，因为很多人一遇到 resume 失败就盯初始化代码，却忽略了问题其实出在 suspend 前没收干净。

## 一句经验总结

suspend/resume 的本质，是一次“系统级状态收敛与重建”。  
高级驱动工程师真正要掌握的，不是会填几个 PM 回调，而是能把在途事务、硬件上下文和恢复顺序设计成一套对称、可重复验证的流程。
