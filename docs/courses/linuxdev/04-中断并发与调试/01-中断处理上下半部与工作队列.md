# 中断处理上下半部与工作队列

## 前言

**C：** 很多驱动一开始都能“跑起来”，但一旦扯上中断，代码就容易突然变得危险起来。原因不在于中断 API 很多，而在于中断上下文的约束非常强：不能乱睡眠，不能做太重的活，稍不留神就会拖垮系统。Linux 为了解决这个问题，逐渐形成了“上半部尽快返回、下半部延后处理”的设计思路。本篇我们就把这条主线讲清楚。

## 上下半部分工图

```mermaid
flowchart LR
  A[IRQ 到来] --> B[上半部 irq_handler]
  B --> C[快速确认与清状态]
  C --> D[schedule_work]
  D --> E[workqueue]
  E --> F[延后处理重活]
```

## 为什么中断处理不能把所有事都写在一个函数里

中断到来时，CPU 需要尽快响应。  
如果中断处理函数里做了太多重活，就会带来几个直接问题：

- 中断关闭或占用时间过长
- 影响系统实时响应
- 容易在错误的上下文里调用会睡眠的接口
- 排查问题时很难区分“中断本身”和“后续处理逻辑”

所以内核里常见的原则是：

::: tip 笔者说
上半部只做必须立即做的事，下半部再处理那些可以延后的工作。
:::

## 什么是上半部

通常所说的上半部，就是最直接的中断处理函数，也就是你通过 `request_irq` 注册进去的 handler。

一个最小中断处理函数大致长这样：

```c
static irqreturn_t ez_irq_handler(int irq, void *dev_id)
{
	/* 1. 读取状态 */
	/* 2. 清中断源 */
	/* 3. 记录必要信息 */

	return IRQ_HANDLED;
}
```

它的特点是：

- 执行快
- 逻辑短
- 尽量不做复杂处理

你可以先把它理解为：**先把火灭了，再说后面的事。**

## 什么是下半部

下半部不是一个单独 API，而是一类设计思路。  
意思是：把中断里不适合立即完成的工作延后处理。

典型可延后的事情包括：

- 较重的数据整理
- 较复杂的状态更新
- 需要和其它线程或子系统协作的逻辑
- 可能需要睡眠的处理

在 Linux 里，常见的下半部实现方式有：

- tasklet
- workqueue
- 线程化中断

本文重点先抓住最容易理解也最常见的 `workqueue`。

## 为什么工作队列很好用

工作队列（workqueue）的核心好处是：

- 它把任务放到一个更适合做“普通工作”的上下文里执行
- 比起硬中断上下文，限制更少
- 对入门者来说，思维负担明显更低

你可以先把它想成：

**中断里先记一笔，然后通知“后台工人线程”来慢慢干。**

## 一个最小示例：中断里只调度 work

下面是一段教学性质的简化示例：

```c
#include <linux/module.h>
#include <linux/interrupt.h>
#include <linux/workqueue.h>

static struct work_struct ez_work;
static int ez_irq = 10; /* 仅作接口形状演示，不建议在未知平台直接照搬加载 */

static void ez_work_handler(struct work_struct *work)
{
	pr_info("ezirq: workqueue handler running\n");
}

static irqreturn_t ez_irq_handler(int irq, void *dev_id)
{
	pr_info("ezirq: irq happened\n");
	schedule_work(&ez_work);
	return IRQ_HANDLED;
}

static int __init ez_init(void)
{
	int ret;

	INIT_WORK(&ez_work, ez_work_handler);

	ret = request_irq(ez_irq, ez_irq_handler, IRQF_SHARED, "ezirq", &ez_irq);
	if (ret) {
		pr_err("ezirq: request_irq failed\n");
		return ret;
	}

	pr_info("ezirq: init success\n");
	return 0;
}

static void __exit ez_exit(void)
{
	free_irq(ez_irq, &ez_irq);
	cancel_work_sync(&ez_work);
	pr_info("ezirq: exit\n");
}

module_init(ez_init);
module_exit(ez_exit);

MODULE_LICENSE("GPL");
```

这段示例的重点不是 IRQ 号本身，而是这条分工：

- `ez_irq_handler`：尽快返回
- `ez_work_handler`：慢一点处理也没关系

::: warning 实验提醒
示例中的 IRQ 号是教学占位值，只适合帮助你理解 `request_irq` 和 `schedule_work` 的配合方式。不要在不了解目标平台 IRQ 分配的情况下，直接把这段代码编译加载到真实机器上做实验。
:::

## `request_irq` 的几个重点

### 第一个参数：IRQ 号

这个值必须来自平台资源，不能乱写。  
示例里写死一个数字只是为了帮助理解接口形状，真正项目里通常来自：

- 设备树
- 平台资源
- 总线枚举结果

### 第三个参数：中断触发和共享标志

常见会看到：

```c
IRQF_SHARED
```

表示该中断线允许共享。  
真实项目是否能共享、该用什么触发方式，要看硬件和平台。

### `free_irq`

申请了中断，就必须在合适时机释放。  
否则模块卸载和设备移除时很容易留下隐患。

## 什么情况下不能在中断里硬做

入门时可以记住这一条粗粒度规则：

- **会阻塞、会睡眠、会很慢** 的事情，不要直接塞进硬中断处理函数

典型容易出问题的场景包括：

- 在中断里拿不合适的锁
- 在中断里做大量日志刷屏
- 在中断里做复杂内存分配
- 在中断里做大段拷贝和繁重计算

## x86 学习环境要注意什么

在普通 PC 或虚拟机里，你并不一定能方便地拿到一个适合教学的真实外设中断源。  
所以中断学习通常会分两步：

1. 先从概念和代码结构上理解
2. 再到具体板级或可控硬件上验证真实中断行为

也就是说，在通用环境里你未必能完整复现某个 SoC 外设的中断时序，但完全可以先把这条原则吃透：

- 硬中断函数尽量短
- 重活放下半部
- 资源申请和释放要成对

## 验证步骤

如果你有可用的中断实验环境，可以按下面思路验证：

1. 驱动加载成功后，确认 `request_irq` 返回 0
2. 触发一次中断
3. 看日志是否依次出现：

```text
ezirq: irq happened
ezirq: workqueue handler running
```

4. 卸载模块时确认：

```bash
sudo rmmod xxx
```

不会卡住，且能正常释放 IRQ 和 work

如果当前环境不方便真实触发 IRQ，也可以先把示例当作代码阅读和结构理解材料。

## 常见问题

### 为什么要先 `cancel_work_sync`

因为 work 可能已经被调度但还没执行完。  
模块退出时如果不等它结束，就可能出现“模块都卸载了，后台工作还在跑”的危险情况。

### `request_irq` 成功是不是就表示硬件中断一定会来

不是。  
这只表示你在内核侧注册成功了，至于硬件是否真的触发，还要结合平台和设备状态判断。

### 下半部是不是只有 workqueue

不是。  
Linux 里下半部形式很多，但对于入门者来说，先把 `workqueue` 这条最容易理解的路线走通，收益最高。

## 小结

中断处理真正难的地方，不是函数原型，而是上下文约束。把“**上半部尽快返回，下半部承接重活**”这条原则建立起来，很多后续设计都会自然顺畅。对于入门驱动开发而言，先会用 `request_irq` 和 `workqueue` 组织最小的中断处理结构，已经是非常关键的一步。
