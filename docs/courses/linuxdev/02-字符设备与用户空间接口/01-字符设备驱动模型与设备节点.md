# 字符设备驱动模型与设备节点

## 前言

**C：** 驱动入门里，字符设备是一条几乎绕不开的主线。原因很简单：它足够典型，又足够直观。用户空间看到的是 `/dev/xxx`，而内核里对应的是设备号、`cdev`、`file_operations` 这一整套模型。把这条链路看清楚，后面再学 `read`、`write`、`ioctl` 就不会只是死记 API。

## 字符设备调用链图

```mermaid

  A[/dev/ezdemo] --> B[设备号 dev_t]
  B --> C[cdev]
  C --> D[file_operations]

```

## 什么是字符设备

字符设备可以粗略理解为：**按字节流方式进行访问的设备**。  
和块设备不同，它通常不强调固定块大小，也不走块层缓存那一套逻辑。

典型例子有：

- 串口设备
- 按键或简单输入输出设备
- 某些传感器的控制接口
- 教学示例里的自定义 `/dev/xxx`

在用户空间，字符设备最常见的访问方式就是：

```c
open("/dev/xxx", O_RDWR);
read(fd, buf, size);
write(fd, buf, size);
ioctl(fd, cmd, arg);
close(fd);
```

## `/dev/xxx` 背后到底是什么

在 Linux 里，设备节点本质上也是文件系统中的一个特殊文件。  
你可以执行：

```bash
ls -l /dev/zero
```

通常会看到类似：

```text
crw-rw-rw- 1 root root 1, 5 ... /dev/zero
```

这里最关键的有两点：

- 开头的 `c` 表示这是一个**字符设备**
- `1, 5` 表示这个设备的**主设备号**和**次设备号**

## 主设备号和次设备号

### 主设备号

主设备号可以理解为：**内核应该把这个设备交给哪类驱动去处理**。

### 次设备号

次设备号可以理解为：**同一类驱动下面的第几个实例**。

举个简单例子：

- 一个驱动可以负责一类传感器
- 具体有 4 个传感器实例
- 那么它们可能共享同一个主设备号，但使用不同的次设备号来区分

所以你可以先记成一句话：

::: tip 笔者说
主设备号决定“谁来管”，次设备号决定“管的是哪一个”。
:::

## 字符设备驱动里的几个核心对象

### `dev_t`

`dev_t` 是内核里用来表示设备号的类型，里面包含了主设备号和次设备号。

常用宏有：

```c
MAJOR(dev)
MINOR(dev)
MKDEV(major, minor)
```

### `struct file_operations`

这是字符设备驱动最核心的结构之一。  
用户空间执行 `open/read/write/ioctl/close`，最终都会落到这里面定义的回调函数上。

一个最小的示意：

```c
static const struct file_operations ez_fops = {
	.owner = THIS_MODULE,
	.open = ez_open,
	.read = ez_read,
	.write = ez_write,
	.release = ez_release,
};
```

### `struct cdev`

`cdev` 可以理解为：**把设备号和 `file_operations` 绑定起来的字符设备对象**。

当内核发现用户空间要访问某个字符设备节点时，最终就是沿着设备号找到这个 `cdev`，再找到对应的操作表。

### `struct inode` 和 `struct file`

在回调里你常会看到这两个参数：

- `struct inode *inode`
- `struct file *filp`

可以先这样理解：

- `inode` 更偏“这个文件本身”
- `file` 更偏“这一次打开行为对应的上下文”

对于入门来说，先知道它们不是一回事就够了。

## 一个字符设备注册流程长什么样

现代字符设备注册，一般使用这一套流程：

1. 申请设备号：`alloc_chrdev_region`
2. 初始化 `cdev`：`cdev_init`
3. 注册 `cdev`：`cdev_add`
4. 创建 class：`class_create`
5. 创建设备节点：`device_create`

::: tip 配套源码
如果你想先看最小可运行骨架，可先阅读 `examples/linuxdev/02-ezpipe/` 目录；它比正文更偏“能编译运行”，而正文更偏“为什么这样设计”。
:::

一个最小骨架如下：

```c
#include <linux/module.h>
#include <linux/fs.h>
#include <linux/cdev.h>
#include <linux/device.h>

#define DEV_NAME "ezdemo"

static dev_t ez_devno;
static struct cdev ez_cdev;
static struct class *ez_class;

static int ez_open(struct inode *inode, struct file *filp)
{
	pr_info("ezdemo: open\n");
	return 0;
}

static int ez_release(struct inode *inode, struct file *filp)
{
	pr_info("ezdemo: release\n");
	return 0;
}

static const struct file_operations ez_fops = {
	.owner = THIS_MODULE,
	.open = ez_open,
	.release = ez_release,
};

static int __init ez_init(void)
{
	int ret;

	ret = alloc_chrdev_region(&ez_devno, 0, 1, DEV_NAME);
	if (ret)
		return ret;

	cdev_init(&ez_cdev, &ez_fops);
	ez_cdev.owner = THIS_MODULE;

	ret = cdev_add(&ez_cdev, ez_devno, 1);
	if (ret) {
		unregister_chrdev_region(ez_devno, 1);
		return ret;
	}

	/*
	 * 新内核常见单参数 class_create()，旧一些的 5.x/LTS 内核
	 * 往往仍需要 class_create(THIS_MODULE, name) 形式。
	 */
#if LINUX_VERSION_CODE >= KERNEL_VERSION(6, 4, 0)
	ez_class = class_create(DEV_NAME "_class");
#else
	ez_class = class_create(THIS_MODULE, DEV_NAME "_class");
#endif
	if (IS_ERR(ez_class)) {
		cdev_del(&ez_cdev);
		unregister_chrdev_region(ez_devno, 1);
		return PTR_ERR(ez_class);
	}

	device_create(ez_class, NULL, ez_devno, NULL, DEV_NAME);

	pr_info("ezdemo: major=%d minor=%d\n", MAJOR(ez_devno), MINOR(ez_devno));
	return 0;
}

static void __exit ez_exit(void)
{
	device_destroy(ez_class, ez_devno);
	class_destroy(ez_class);
	cdev_del(&ez_cdev);
	unregister_chrdev_region(ez_devno, 1);
}

module_init(ez_init);
module_exit(ez_exit);

MODULE_LICENSE("GPL");
```

## 这段流程到底在完成什么

### `alloc_chrdev_region`

向内核动态申请一段设备号。  
相比手动指定主设备号，这种方式更适合入门和一般驱动开发。

### `cdev_init` + `cdev_add`

把你的操作集和设备号挂到一起，让内核知道：

- 这个设备号归谁管
- 当用户态访问时该调用哪些回调

### `class_create` + `device_create`

这一层主要是为了让用户空间更自然地看到 `/dev/ezdemo`。  
否则即使你内核侧已经注册成功，用户态也不一定直接能用。

::: tip 版本说明
如果你用的是 5.x 或较老的 LTS 内核，`class_create` 很可能仍需要 `THIS_MODULE` 作为第一个参数；若你照着示例编译却在这里报错，优先先看目标内核版本，再按对应原型调整。
:::

## 字符设备和 `/proc/devices`

当字符设备注册成功后，可以查看：

```bash
cat /proc/devices
```

这里能看到当前系统里已注册的字符设备主设备号信息。  
如果你的驱动名字正确出现在里面，说明字符设备注册至少走通了一半。

## 验证步骤

建议做下面几步实验：

1. 编译并加载示例模块。
2. 查看日志：

```bash
sudo insmod ezdemo.ko
dmesg -T | tail -n 20
```

3. 查看 `/proc/devices`：

```bash
cat /proc/devices | grep ezdemo
```

4. 查看 `/dev` 节点：

```bash
ls -l /dev/ezdemo
```

5. 尝试简单打开它：

```bash
python3 - <<'PY'
import os
fd = os.open("/dev/ezdemo", os.O_RDWR)
os.close(fd)
print("open ok")
PY
```

## 常见问题

### 加载模块成功，但没有 `/dev/ezdemo`

优先检查：

- `class_create` / `device_create` 是否执行成功
- 系统是否正确挂载了 `devtmpfs`
- 设备号是否注册成功

### `/proc/devices` 里有名字，但 `open` 失败

这种情况下，要继续确认：

- `/dev/ezdemo` 节点是否存在
- 节点对应的主次设备号是否正确
- 你的驱动回调是否真的挂上去了

### 为什么不直接手工 `mknod`

可以手工创建，但教学和日常开发中，优先掌握 `class_create` + `device_create` 更自然。  
手工 `mknod` 更像排障时的辅助验证手段。

## 小结

字符设备驱动的核心，不是某一个 API，而是一整条链路：**设备号 -> `cdev` -> `file_operations` -> `/dev/xxx` -> 用户空间系统调用**。把这条链路搞明白后，接下来去实现 `open/read/write/release` 时，你就知道自己不是在“写四个孤立函数”，而是在补全整个字符设备模型的关键行为。
