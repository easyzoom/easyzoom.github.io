# 驱动分层、平台差异隔离与BSP适配策略

## 前言

**C：** 很多驱动在最初 bring-up 阶段都能“先跑起来”，但一旦同时面对多个 SoC、多个板型、多个内核版本，就会迅速失控。寄存器差异、时钟复位差异、DMA 路径差异、quirk 处理到处散落，最终 nobody wants to maintain it。高级驱动工程师真正重要的能力之一，就是把平台差异收敛进清晰边界，让驱动既能跟硬件走，又不至于被 BSP 绑死。

## 分层的核心目标

```mermaid
flowchart LR
  hwSpecific["平台/板级差异"] --> adaptLayer["适配层或quirk层"]
  adaptLayer --> coreLogic["核心驱动逻辑"]
  coreLogic --> subsystemFace["对子系统/上层统一接口"]
```

## 为什么驱动一长就开始难维护

常见原因不是功能太多，而是差异没有被隔离。  
例如这些东西混在一起：

- 核心状态机
- 板级 GPIO/clock/reset 差异
- SoC 特定 workaround
- 调试逻辑
- PM 和恢复路径

这样一来，每次改一个平台问题，都可能误伤另一个平台。

## 分层不是为了好看，而是为了控制变更影响面

一个更成熟的驱动通常会努力把下面三层分开：

1. **核心功能层**  
   设备真正的业务状态机、队列、协议和错误处理。
2. **平台适配层**  
   clock/reset/regulator、DMA 限制、quirk、板级差异。
3. **框架接入层**  
   与 netdev、input、V4L2、字符设备等子系统的接口。

只要这三层边界清楚，大部分变更都能局部化。

## BSP 适配最怕什么

最怕“为了让某块板先跑起来”，把大量临时判断直接塞进核心路径。  
例如：

- 到处都是 if (soc_is_xxx())
- 板级时序特殊处理插进通用读写逻辑
- 平台 workaround 混到热路径

短期能救火，长期会把驱动结构拖垮。

## 更稳妥的做法

高级工程师通常会尽量把差异收敛成：

- 平台数据
- ops 表
- quirk 标志
- 资源封装函数

这样核心逻辑只看到“能力”或“行为差异”，而不是到处感知具体平台名字。

## 什么时候应该抽适配层

一旦出现以下迹象，就该考虑抽象：

- 第二个平台接入时改动面很大
- PM/恢复路径开始出现平台特判
- DMA/中断/时钟能力差异开始影响核心状态机
- 测试矩阵扩大后，一个修复经常破另一个平台

## 一句经验总结

驱动分层的价值，不在于代码看起来更“优雅”，而在于让平台差异不会污染核心状态机。  
高级驱动工程师写驱动时，脑子里始终要有一句话：哪些是设备本质，哪些只是平台偶然性。
