# netdev、DRM、V4L2框架入门方法论

## 前言

**C：** 真正进入高级驱动工程阶段后，你会发现很多重要设备已经不是“写一个 probe + 几个寄存器函数”那么简单了。网络设备会走 `netdev`，显示会进 `DRM/KMS`，摄像头和视频链路会进 `V4L2`。这些框架不是“更复杂的字符设备”，而是完整的子系统生态。面对这类框架，最重要的不是一开始就啃完所有代码，而是掌握一套正确的入门方法论。

<!-- more -->

## 面对大型子系统的正确姿势

```mermaid
flowchart LR
  hwModel["先理解硬件模型"] --> subsystemRole["再理解子系统抽象"]
  subsystemRole --> minimalPath["找到最小可运行路径"]
  minimalPath --> observe["观察数据流与回调关系"]
  observe --> extend["最后再做性能和功能扩展"]
```

## 不要从“所有源码”开始

很多工程师第一次看 netdev/DRM/V4L2，会犯一个常见错误：  
试图从全量源码和所有回调一口气理解完整框架。

这通常会直接把自己淹没。  
更好的方法是：

1. 先理解硬件提供什么能力
2. 再理解框架要求你扮演什么角色
3. 找最小路径把一条完整功能链走通

换句话说，不是先看“框架有多少文件”，而是先看“我的设备在这个框架里是什么对象”。

## 先理解“框架想统一什么”

### `netdev`

统一的是网络设备收发、队列、统计、链路状态与协议栈协作。

### `DRM/KMS`

统一的是显示 pipeline、buffer、plane、connector、mode setting 等图形显示抽象。

### `V4L2`

统一的是视频设备、buffer 队列、格式协商、流控制和媒体链路。

所以学习这些框架时，第一步不是 API，而是：  
**框架在替整个系统统一什么语义。**

## 再找“最小可运行路径”

高级工程师面对复杂框架时，常会主动找最小闭环，例如：

- netdev：链路拉起、发一个包、收一个包
- DRM：点亮一个最小显示路径
- V4L2：拿到一帧有效 buffer

只要最小闭环清楚，很多回调和状态机就会逐渐有位置感。  
否则你会一直停留在“这个回调看起来也很重要”的信息过载状态。

## 数据流比 API 列表更重要

在这类大型子系统里，真正该理解的是：

- 数据从哪来
- 经谁排队
- 谁触发回调
- 谁负责回收
- 错误或断流时由谁恢复

这套数据流意识，比背几十个 API 名字更重要。  
因为大多数框架 bug，最后都落在数据流边界或生命周期错位上。

## 为什么高级工程师更重视参考驱动

面对大型框架时，参考同类上游驱动往往极其重要。  
原因不是“抄代码”，而是看：

- 这种设备在框架里一般怎么建模
- 关键回调边界怎么划分
- 热路径和慢路径怎么拆
- PM、恢复、错误处理怎么接进去

这能显著减少“自己凭感觉设计框架接入层”的风险。

## 一些重要的自我约束

1. 不要先写私有接口，再想着以后接框架
2. 不要为了短期 bring-up 破坏框架语义
3. 不要跳过最小闭环，直接堆功能
4. 不要只看成功路径，不看 buffer 回收和错误恢复

## 一句经验总结

面对 `netdev`、`DRM`、`V4L2` 这类大型子系统时，最重要的是方法论：  
先理解硬件模型，再理解框架抽象，先跑通最小闭环，再逐步扩展。高级驱动工程师和普通“能改两行 bring-up”的差距，往往就体现在这里。
